# Versioning

Every merge to `main` runs `semantic-release` per package. There is
**no Release PR** and no manual version bump — the commit convention
drives the version bump directly.

## Commit → version bump

| Commit type              | Bumps       |
|--------------------------|-------------|
| `feat(scope):`           | **minor**   |
| `fix(scope):`            | **patch**   |
| `perf(scope):`           | **patch**   |
| `BREAKING CHANGE:` (footer) | **major** |
| `chore` / `docs` / `test` / `refactor` / `ci` / `build` | none (silent no-op) |

## Per-package versioning

Each package has its own `release.config.js` with `commitPaths` scoped to
its directory. A merge only cuts a release for packages whose files
actually changed since the last tag.

Tag format includes the package name so multiple packages can coexist:

```
@rn-ds/tokens-v0.3.1
@rn-ds/components-v1.0.0
@rn-ds/icons-v0.2.0
```

## The [skip ci] loop breaker

After semantic-release publishes, it commits `package.json` + `CHANGELOG.md`
back to `main` with `[skip ci]` in the commit message. This prevents the
release commit from re-triggering the release workflow. Do not remove the
`[skip ci]` filter in `release.yaml`.

## Snapshot channel

For WIP consumption without a real release:

```bash
gh workflow run release.yaml -f publish_snapshot=true
```

Publishes `0.0.0-<sha>.tgz` tarballs as workflow-run artifacts. Consumers
install directly from those URLs.
